mostly making things that were silently broken behave correctly.

## Research page (RDW / MNTS bug)

### Symptom

User submitted research for RDW + MNTS today, the UI showed the entry
as completed, but expanding it revealed nothing — no report body, no
trade suggestions, no error.

### Root cause

`supabase/migrations/20260506_ai_transparency.sql` adds five columns
across three tables. It was applied to `csp_scans` and
`daily_recommendations` but **not** to `research_reports`, which kept
the old schema (no `ai_provider`, `ai_model`, `error_message`).

`completeReport()` and `failReport()` in `src/lib/db/research.ts`
unconditionally write to those missing columns. Postgres rejects the
UPDATE, the `.update().eq().select().single()` call throws, the POST
handler's outer try/catch swallows it as a 500, but the row stays at
`status=running` with `report=''` forever.

The history endpoint then returns all rows (no status filter), and the
research page **hardcoded** `status: "completed"` on every history row
on the way in. Result: zombie running rows masquerade as completed
empty reports.

### Fix

1. Applied the missing columns to `research_reports` via the Supabase
   MCP (`research_reports_ai_metadata_backfill`).
2. Cleared the 4 zombie rows: RDW/MNTS, LLY, SMR, NVDA — all set to
   `status='failed'` with an `error_message` explaining the cause.
3. `src/app/api/research/history/route.ts` now passes `status`, `mode`,
   `aiProvider`, `aiModel`, and `error` through to the client.
4. `src/app/(app)/research/page.tsx` respects the returned `status`
   instead of forcing it to `"completed"`.
5. `EntryCard` now shows a "Re-run research →" hint when a completed
   row has neither report text nor suggestions, so this failure mode
   isn't silent in the future.

## Portfolio Manager

### Symptom

`portfolio_manager_scans` shows nothing but failures since June 1, all
with the same shape:

```
Expected ',' or ']' after array element in JSON at position 28623
```

…with the position always landing in the 27k–29k character range.

### Root cause

`src/lib/portfolio-manager/analyst.ts` calls `generateText` with
`maxTokens: 8000`. For 23–25 tickers, the JSON `analyses` array
overflows that cap mid-string. The model gets cut off, the response
parser can't recover, the scan fails.

### Fix

- Raised `maxTokens` from 8000 → 16000. With Sonnet 4.5's 64k output
  cap there's plenty of headroom; 16k gives ~700 tokens/ticker.
- Added `repairTruncatedJson()` in the analyst's `extractJson` path:
  walks the candidate string tracking string/escape state and
  brace/bracket depth, then either (a) appends the closers needed to
  balance the depth stack, or (b) walks back to the last `,` at
  array-depth and closes there. So even if the response gets truncated
  again, we recover a partial result instead of failing the whole scan.

### Cron schedule

Added a market-close run alongside the existing open run in
`vercel.json`:

```json
{ "path": "/api/cron/portfolio-manager?trigger=open",  "schedule": "30 13 * * 1-5" }
{ "path": "/api/cron/portfolio-manager?trigger=close", "schedule": "5 20 * * 1-5" }
```

`5 20 UTC` = 16:05 ET = 5 minutes after the bell, weekdays. The route
already reads `trigger=open|close` and stamps the DB row with
`cron-open` or `cron-close` so we can tell them apart in history.
