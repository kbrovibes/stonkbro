# v0.23.0 ÔÇö Dark Mode

## Summary
Robinhood-style dark mode across the entire app with a Light/Dark/Auto toggle in the profile dropdown. Dark is the new default for users who haven't expressed a preference; OS scheme is honored only when the user explicitly picks "Auto". Light mode is byte-identical to v0.22 by construction ÔÇö every dark annotation is additive next to its light counterpart, and the semantic token light values map 1:1 to the prior hardcoded values.

## Notes

- **Light mode unchanged.** All light-mode token values match the corresponding stone-/gray-/sky-/emerald-/red- Tailwind shade exactly. Every new annotation is additive (`dark:` prefix), never replacing the light class. A user on Light will see pixel-identical output to v0.22.
- **Migration script is idempotent.** `node scripts/add-dark-variants.mjs` skips any class already paired with a `dark:` companion. Re-run after merges that introduce new components with hardcoded `bg-white` / `text-stone-*`.
- **CLI-bug workaround for env vars.** During preview-deployment validation, the `vercel env add --value` CLI silently stored empty values for branch-scoped Preview entries. Worked around via Vercel REST API (`POST /v10/projects/{id}/env`). Not relevant after merge to main ÔÇö production env is the source of truth.
- **PWA manifest dark variant deferred.** `public/manifest.json` has only one `background_color` / `theme_color`. The spec doesn't support dark variants. iOS PWA uses the dynamic `<meta name="theme-color">` which already flips; Android PWA splash will be light briefly on first launch. Acceptable.
- **Charts that still need polish.** The amber cash bars and teal RSU bars in `/time-machine/detail`, and the amber collateral fills in `/portfolio`, stayed as hardcoded hex on purpose ÔÇö they pop on both backgrounds. If any read poorly, swap for `var(--amber-strong)` style tokens in a follow-up.
