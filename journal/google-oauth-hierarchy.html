Google OAuth Hierarchy

How Google Cloud projects, OAuth clients, and consent screens relate to each other

Structure

Google Account └── Google Cloud Project (e.g. "My Apps") ├── OAuth Consent Screen ← ONE per project │ ├── App Name "Stonkbro" — what users see on the picker │ ├── App Logo │ ├── Homepage URL https://stonkbro.app │ └── Authorized Domains stonkbro.app, supabase.co │ └── OAuth 2.0 Clients ← MANY per project, all share consent screen above ├── Client A (Web) │ ├── Client ID 643285704755-brm9...apps.googleusercontent.com │ ├── Client Secret GOCSPX-... │ ├── Authorized JS Origins https://stonkbro.app │ └── Authorized Redirect URIs https://enbevnx...supabase.co/auth/v1/callback ├── Client B (iOS app) └── Client C (Android app)

Why you see enbevnxmowcqdypqyyio.supabase.co Google shows the redirect URI domain when the consent screen app name is unverified. Verification requires proving ownership of a domain — and since you don't own supabase.co, Google falls back to displaying the raw Supabase subdomain.

Fix Options

Easiest
Brand the consent screen + verify stonkbro.app

5 min
Separate Google Cloud project per app
Create a new project for stonkbro with its own consent screen. Fully independent branding, quota, and credential management.
Supabase Pro
Supabase custom auth domain
Set up auth.stonkbro.app as a custom Supabase auth domain. The redirect URI then shows your domain instead of Supabase's — fully branded end-to-end. Requires Supabase Pro plan.
